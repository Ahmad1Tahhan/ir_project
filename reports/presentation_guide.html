  • صفحة التوثيق التفاعلية (الأسهل): كل خدمة FastAPI تنشر واجهة Swagger تلقائياً على المسار /docs. افتح مثلاً http://127.0.0.1:8002/docs ← اختر الـ endpoint ← اضغط Try it out ← أدخل القيم ← Execute. هذا أنظف من curl ويُظهر الطلب والرد مباشرة.

🧭 ترتيب العرض (تسلسل الكلام بين الأعضاء)

اعرضوا بنفس ترتيب رحلة الاستعلام داخل النظام — فيصير العرض قصة متصلة:

1 — احمد الطحانمقدمة + المعالجة المسبقة + بوابة API 2 — محمد دريد فاخوريTF-IDF / BM25 + الفهرس + تحسين الاستعلام 3 — طارق المزيكEmbeddings + FAISS + الهجين 4 — ياسر جمال الدينالتقييم + الرسومات + النوتبوك 5 — معتز الحلاقالميزات الإضافية + RAG + الواجهة

🔧 ملاحظة على الواجهة: في الشريط الجانبي مفتاح Backend فيه خياران: Direct (افتراضي — يُحمّل المحرك داخل عملية الواجهة، يعمل أوفلاين بلا خدمات) وGateway (SOA) (يستدعي الخدمات عبر البوابة). لإظهار بنية SOA بدّل إلى Gateway (SOA)؛ وعند أي مشكلة شبكة ارجع إلى Direct. تبويب RAG يستدعي خدمة RAG مباشرة في الوضعين.

عضو 1 — احمد الطحان

1 احمد الطحان

المسؤولية: المقدمة العامة + المعالجة المسبقة (Pre-Processing) + بوابة API (Gateway)
🗣️ ماذا تقول (أنت من يفتح العرض)
«مشروعنا محرك بحث متكامل مبني وفق مبدأ SOA بلغة Python فقط، يعمل على مجموعة Quora التي تحوي 522,931 وثيقة و10,000 استعلام مع ملف qrels للتقييم، ويعمل كاملاً بدون إنترنت. سأبدأ بالمعالجة المسبقة وهي أول خطوة في رحلة أي استعلام، ثم بوابة الـ API التي تربط الخدمات.»
🖥️ خطوات العرض العملي
  1. افتح صفحة خدمة المعالجة: http://127.0.0.1:8002/docs ← endpoint POST /preprocess ← Try it out، وأدخل: { "text": "The Studying of Running CATS!! http://x.com", "normalizer": "lemmatize" } نفّذ وأظهر الخرج: "tokens": ["studying","running","cat"]. 🔧 كيف نفّذناها (في preprocessing/text.py): السلسلة هي تحويل لأحرف صغيرة ← إزالة الروابط والرموز غير ASCII بتعبير نمطي (Regex) ← تقطيع بتعبير نمطي يُبقي الفواصل العليا ← إسقاط الكلمات الأقصر من حرفين ← إزالة كلمات التوقف (قائمة NLTK) ← Lemmatization عبر WordNetLemmatizer. لذلك «CATS» تصبح «cat» و«Running» تبقى «running» وتُحذف «The/of».
  2. غيّر normalizer إلى "stem" ثم "none" وأعد التنفيذ لتُظهر أن السلسلة قابلة للتبديل. 🔧 كيف: الإعداد PreprocessConfig.normalizer يختار بين WordNetLemmatizer و PorterStemmer أو بلا تطبيع — ويُحفظ مع المخرجات حتى يُعالَج الاستعلام وقت البحث بنفس الطريقة التي عُولجت بها الوثائق (شرط أساسي للتوافق).
  3. انتقل لبوابة الـ API: افتح http://127.0.0.1:8000/health وأظهر أن الخدمات حالتها ok. 🔧 كيف نفّذناها (في services/gateway.py): البوابة تستعلم صحّة كل من retrieval/preprocessing/refinement/evaluation وتُرجِع حالتها مجمّعة. (خدمة RAG تُفحص مستقلة على :8005/health لأنها اختيارية وثقيلة.)
  4. افتح http://127.0.0.1:8000/docs وأظهر مسارات البوابة (/search, /preprocess, /refine, /report ...). 🔧 كيف: البوابة لا تكرّر المنطق، بل تمرّر كل طلب عبر requests إلى الخدمة المناسبة (REST). هذا يحقّق Loose Coupling: الواجهة تكلّم عنواناً واحداً ولا تعرف أماكن الخدمات.

س: لماذا Lemmatization وليس Stemming؟
ج: الـ Lemmatization يُرجع جذراً صحيحاً لغوياً (cats→cat) فيحافظ على المعنى؛ جعلناه قابلاً للتبديل إلى Stemming الأسرع لإتاحة المقارنة.

س: لماذا نفس السلسلة على الوثائق والاستعلام؟
ج: حتى يكون تمثيل الاستعلام مقارَناً لتمثيل الوثائق؛ لذلك نحفظ إعدادات المعالجة مع المخرجات ونعيد تحميلها وقت البحث.

س: ما فائدة بوابة الـ API؟
ج: نقطة دخول موحّدة تُخفي عناوين الخدمات وتفصل الواجهة عنها؛ وكل خدمة تبقى قابلة للتشغيل والاختبار وحدها.

عضو 2 — محمد دريد فاخوري

2 محمد دريد فاخوري

المسؤولية: تمثيل TF-IDF و BM25 + الفهرس المعكوس + تحسين الاستعلام (Refinement)
🗣️ ماذا تقول
«بعد المعالجة نحتاج تمثيلاً رقمياً للوثائق. بنيتُ تمثيلين كلاسيكيين: VSM TF-IDF مع جيب التمام، و BM25 فوق فهرس معكوس صنعناه يدوياً — وميزته أننا نحسب الدرجة وقت الاستعلام فنستطيع تغيير معاملاته k1 و b حياً. كما أضفتُ خدمة تحسين الاستعلام.»
🖥️ خطوات العرض العملي (في تبويب Search بالواجهة)
  1. اختر من قائمة Representation / method الخيار TF-IDF (VSM)، واكتب استعلاماً مثل how do i learn python programming، ونفّذ. أظهر النتائج بالنص الأصلي + المُعرّف doc_id + الدرجة. 🔧 كيف نفّذناها (representation/tfidf.py): نستخدم TfidfVectorizer من scikit-learn مع مُقطّع هوية (identity tokenizer) فوق الرموز المعالَجة مسبقاً، والمطابقة بجيب التمام (Cosine).
  2. بدّل الطريقة إلى BM25 (probabilistic) ونفّذ نفس الاستعلام؛ قارن الترتيب. 🔧 كيف نفّذناها (indexing/inverted_index.py): بنينا فهرساً معكوساً يدوياً يخزّن قوائم الإحالة (postings) وترددات الكلمات بصيغة مصفوفات NumPy مضغوطة (CSR) لسرعة الوصول.
  3. أهم عرض: افتح قسم Parameters، وحرّك منزلقَي BM25 k1 وBM25 b ثم أعد البحث — أظهر تغيّر الترتيب حياً. 🔧 كيف: درجة BM25 لا تُحسب مسبقاً بل وقت الاستعلام فوق الفهرس المعكوس بدلالة k1 و b، فيمكن تمرير قيم جديدة لكل بحث ومشاهدة الأثر مباشرة. القيم الافتراضية k1=1.5 و b=0.75.
  4. فعّل صندوق Show query refinement suggestions واكتب استعلاماً فيه خطأ إملائي مثل pyton programing language، وأظهر التصحيح المقترح والاستعلام المُحسّن. 🔧 كيف نفّذناها (refinement/query_refine.py): تصحيح إملائي للكلمات خارج مفردات الفهرس بأقرب كلمة عبر مسافة التحرير (difflib بعتبة 0.85)، وتوسيع بالمرادفات من WordNet، واقتراحات من سجلّ استعلامات المجموعة نفسها.
  5. (اختياري) اختبر الخدمة وحدها: http://127.0.0.1:8003/docs ← POST /refine بالجسم: { "dataset": "quora", "query": "pyton programing" } وأظهر الحقول spelling_changes وrefined_query. ولاقتراحات الإكمال: GET /suggest?prefix=how.

س: ماذا يعني k1 و b؟
ج: k1 يضبط تشبّع تكرار الكلمة (كم يفيد التكرار الإضافي)، و b يضبط تطبيع طول الوثيقة (0 = تجاهل الطول، 1 = تطبيع كامل).

س: لماذا فهرس معكوس يدوي بدل مكتبة جاهزة؟
ج: لتحقيق متطلب بناء الفهرسة بأنفسنا، ولأنه يتيح حساب الدرجة وقت الاستعلام وتغيير المعاملات حياً.

س: هل التصحيح الإملائي يضمن إصلاح كل خطأ؟
ج: لا؛ يصحّح فقط إن وُجدت كلمة في مفردات الفهرس قريبة بما يكفي (عتبة 0.85)، وإلا يُبقي الكلمة كما هي.

عضو 3 — طارق المزيك

3 طارق المزيك

المسؤولية: التمثيل الدلالي (Embeddings) + مخزن المتجهات FAISS + التمثيل الهجين + خدمة Retrieval
🗣️ ماذا تقول
«التمثيلات الكلاسيكية تطابق الكلمات حرفياً، لكن أسئلة Quora متشابهة دلالياً لا لفظياً. لذلك أضفتُ تمثيلاً دلالياً بنموذج all-MiniLM-L6-v2 يعمل على الـ GPU، نخزّن متجهاته في FAISS للبحث السريع، ثم نجمع الأفضل من العالمين بالتمثيل الهجين.»
🖥️ خطوات العرض العملي (تبويب Search)
  1. اختر Embeddings (semantic) ونفّذ استعلاماً «دلالياً» لا تتطابق ألفاظه مع الجواب، مثل: how can I earn money from home — أظهر أنه يجد أسئلة عن الدخل عبر الإنترنت رغم اختلاف الكلمات. 🔧 كيف نفّذناها (representation/embeddings.py): نُرمّز كل وثيقة إلى متجه 384-بُعد بنموذج sentence-transformers على الـ GPU، ونطبّع المتجهات (L2) فيصبح الضرب الداخلي = جيب التمام.
  2. اشرح مخزن المتجهات: نخزّن المتجهات في فهرس FAISS من نوع IndexFlatIP للبحث السريع على نصف مليون وثيقة. 🔧 كيف نفّذناها (indexing/vector_store.py): IndexFlatIP يبحث بالضرب الداخلي فوق متجهات مطبّعة، فيعطي أعلى الوثائق تشابهاً دلالياً مع الاستعلام.
  3. اختر Hybrid — Serial (BM25 → rerank) ونفّذ، ثم Hybrid — Parallel (fusion) وقارن. 🔧 كيف نفّذناها (retrieval/hybrid.py): التسلسلي يأخذ مرشّحي BM25 (حجمهم = «Candidate pool») ثم يعيد ترتيبهم بجيب تمام الـ Embeddings — أدق وأرخص. التفرّعي يشغّل BM25 والـ Embeddings مستقلين ثم يدمج قائمتيهما.
  4. في وضع Parallel غيّر Parallel fusion بين rrf وweighted وأعد البحث. 🔧 كيف نفّذناها (retrieval/fusion.py): RRF يدمج بجمع مقلوب الرُّتب Σ 1/(60+rank) فلا يحتاج معايرة درجات بين مقاييس مختلفة؛ وWeighted يطبّع درجات كل قائمة (min-max) ثم يجمعها بأوزان.
  5. اذكر التسريع: ترميز كل الوثائق على 4060 يستغرق نحو دقيقة، وبناء كل المخرجات نحو 80 ثانية.
  6. (اختياري) اختبر خدمتك وحدها: http://127.0.0.1:8001/docs ← POST /search بالجسم: { "dataset":"quora", "method":"embedding", "query":"learn python", "top_k":5 } وكذلك GET /methods لعرض الطرق المتاحة.

س: لماذا تطبيع المتجهات قبل FAISS؟

س: ما الـ Candidate pool؟
ج: عدد المرشّحين الذين يولّدهم BM25 قبل إعادة الترتيب الدلالي في الوضع الهجين — كلّما زاد زاد الاستذكار مقابل بطء أكبر.
